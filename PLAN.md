# ZenAir КП — план разработки

Внутренний рабочий план. Учебный дневник ведётся отдельно в `PROGRESS.md`.

## Что строим

Mobile-first веб-приложение для менеджеров zenair.by. Сценарий: клиент звонит → менеджер
на телефоне выбирает кондиционер(ы), отмечает нужные монтажные работы, вбивает имя/телефон
клиента и скидку → жмёт «Печать» → получает PNG или PDF готового КП и отправляет в мессенджер.

Референс — существующее приложение на Angular 18 + Laravel (`iphon`). Переписываем с нуля,
исправляя его архитектурные проблемы (см. «Чего не повторяем»).

## Данные WooCommerce

Магазин — репозиторий `G:\server\claude\zenair` (БД `dzenairby_cond`, ~76 опубликованных товаров).

Доступные атрибуты товара:

| Таксономия | Смысл | В КП |
|---|---|---|
| `product_brand` | бренд + серия (напр. `gree`, `pular`) | Бренд → Серия |
| `pa_area` | площадь помещения (`do-25`, `25-35`) | да |
| `pa_noise_inner` | уровень шума, дБ | да |
| `pa_inverter` | инвертор да/нет | да |
| `pa_min_temp` | мин. температура работы | да |
| `pa_wi_fi` | Wi-Fi | да |
| `pa_class_energy` | класс энергоэффективности | да |
| `pa_type_system` | тип системы | фильтр |
| мета `model_power_kw` | мощность на охлаждение, кВт (напр. `2.50 (0.50 ~ 3.25)`) | да |

Мощность на охлаждение в кВт добавлена в WooCommerce как произвольное поле
`model_power_kw` (обновлено 2026-07-15) — синкаем в `Product.attributes` и **показываем**
в карточке товара и КП.

Каскад выбора: **Бренд → Серия → Модель** строится из `product_brand`
(первый терм = бренд, второй = серия) + название товара.

Синк: эндпоинт `POST /admin/sync-products` тянет WooCommerce REST API
(`/wp-json/wc/v3/products`, ключи `ck_`/`cs_` в `.env`) и upsert-ит в `Product` по `wcId`.
Цена копируется в `price`, но остаётся редактируемой в приложении.

## Модель данных (Prisma)

- `Manager` — id, email, passwordHash, name, createdAt
- `Product` — id, wcId, sku, name, brand, series, price, imageUrl, attributes (Json), active
- `InstallationWork` — id, code, title, defaultPrice, sortOrder (справочник работ)
- `Client` — id, name, phone, address
- `Quote` — снепшот КП: managerId, clientId, items (Json), works (Json), discount, prepayment,
  guarantee, delivery, dayWorks, total, createdAt

`Quote` хранит снепшот, а не ссылки: цены товаров и работ меняются, а выданное клиенту КП
должно оставаться воспроизводимым.

## Дефолты монтажных работ (из референса, BYN)

| Работа | Цена |
|---|---|
| Монтаж оборудования (базовый, × кол-во кондиционеров) | 450 |
| Штробление стен | 80 |
| Монтаж коммуникаций | 760 |
| Монтаж дренажного насоса | 280 |
| Установка козырька | 305 |
| Установка корзины | 311 |
| Монтаж кронштейна | 316 |
| Демонтаж сплит-системы | 338 |
| Демонтаж внутреннего блока | 344 |
| Демонтаж наружного блока | 349 |

Формула итога:

```
товары       = Σ(price × qty) по всем добавленным кондиционерам
со скидкой   = floor(товары × (1 − discount/100))
базовый монтаж = Σ(qty кондиционеров) × basePrice(450)
работы       = Σ(price × qty) по выбранным работам
ИТОГО        = со скидкой + базовый монтаж + работы
```

Скидка применяется **только к оборудованию**, не к монтажу — так в референсе.

## Дефолты футера КП

Гарантия на оборудование 36 мес, на монтаж 12 мес (константа), срок поставки 3 раб. дня,
срок работ 7 дней, доставка бесплатно со склада. Первые — редактируются в правой панели.

## Этапы

Один законченный этап за раз. После каждого — резюме заказчику + запись в `PROGRESS.md`.

- **0. Инициализация** — монорепо, Vite+React+TS+Tailwind, Express+TS, Prisma (`Manager`),
  ESLint+Prettier, Vitest + Playwright со smoke-тестом на каждый.
- **1. Авторизация** — `/login` с табами Авторизация/Регистрация, RHF+Zod с ошибками под полем,
  `POST /auth/register`, `POST /auth/login`, bcrypt, JWT access+refresh, защищённый роут.
  Тесты: Playwright — успешный/неуспешный логин; Vitest — валидация формы.
- **2. Каркас главного экрана** — верхнее меню (логотип по центру, «Монтаж» слева / «Заказчик»
  справа), header компании, заглушка КП, нижнее меню (Сброс / Печать с выбором PNG-PDF / Выход
  с подтверждением). Панели на `useState`, данных пока нет.
- **3. Выбор кондиционера** — модель `Product`, синк из WooCommerce + сид на 5–10 моделей,
  `GET /products` с фильтрами, каскадный выбор Бренд→Серия→Модель, **список товаров**
  (не 3 фиксированных слота!), у каждого своё количество.
- **4. Стоимость и монтаж** — модель `InstallationWork` + сид дефолтами, левая панель
  редактирования количества/цены работ, автопересчёт итога, скидка %.
  Здесь же решаем, где живёт стейт КП (склоняюсь к одному контексту/стору `QuoteProvider`).
- **5. Доп. функции** — правая панель «Заказчик» (скидка, предоплата, баннер, сроки),
  блок баннера в КП, «Сброс» чистит данные заказчика но сохраняет монтаж, футер КП.
- **6. Данные клиента** — модели `Client` и `Quote`, форма имя/телефон/адрес с валидацией,
  модалка при попытке печати с пустыми имя/телефон, `POST /quotes` при генерации файла.
- **7. Генерация КП** — компонент финального документа, PNG через html-to-image,
  PDF через @react-pdf/renderer, скачивание. Playwright: логин → печать.
- **8. Полировка** — полный E2E-путь менеджера, мобильный вьюпорт (iPhone SE), accessibility
  (лейблы, контраст), финальное ревью `PROGRESS.md`.

## Чего не повторяем из референса

Референс работает, но написан тяжело. Осознанно делаем иначе:

1. **`product1/2/3` как отдельные компоненты-близнецы** → один `<ProductRow>` и массив товаров
   в сторе. Иначе четвёртый кондиционер требует копипасты целого компонента.
2. **~40 плоских полей `customerDrill`, `customerDrillPrice`, `customerDrillQuantity`…** →
   массив `works: {code, title, price, qty}[]`. Добавить работу = добавить строку в сид.
3. **Фильтрация брендов по хардкоду email** (`validEmails: ['dmitrii.yakovlev@mail.ru', …]`) →
   если разграничение доступа нужно, это поле роли у `Manager`, а не список в коде фронта.
4. **Зоопарк `BehaviorSubject` в 5 сервисах** → один источник правды по КП.
5. **PNG/PDF через html2canvas по всему `<app-home>`** (тянет в картинку меню и панели) →
   отдельный компонент документа, рендерим только его.
